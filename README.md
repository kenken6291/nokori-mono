## ① レスポンシブ対応の考え方（index.html の解説）

JSでデバイスを判定するのではなく、**CSSが画面幅に応じて自動最適化する**「モバイルファースト」構成です。UA文字列によるデバイス判定は誤判定・将来の破綻が多いため使いません（JSで分岐が必要な場面だけ `window.matchMedia` を使用。コード末尾に例あり）。

要点（コード内に ★コメントで対応箇所を明記しています）:

1. **食材グリッドの自動列数**
   `grid-template-columns: repeat(auto-fill, minmax(min(96px, 28vw), 1fr))`
   列数を固定しないため、スマホで3〜4列、タブレットで5〜6列、PCで6列以上に勝手に調整されます。ブレークポイントを増やさなくても破綻しません。

2. **「きょうのまな板」の二刀流レイアウト**
   - スマホ（〜959px）: 画面下に**固定バー**として常時表示。親指が届く位置に「まな板＋検索ボタン」をまとめ、`env(safe-area-inset-bottom)` で iPhone のホームバーと重ならないようにしています。本文側には `padding-bottom` を入れてバーに隠れないようにしています。
   - PC（960px〜）: 固定バーをやめ、**右サイドの sticky パネル**に切替。`max-width: 1100px` のコンテナで横に間延びするのを防ぎます。

3. **タップしやすさ**
   - 全ボタンに `min-height: 48px`（Apple/Google の推奨タップ領域）
   - `touch-action: manipulation` でダブルタップズームを抑止し連打可能に
   - `:active` で押し込みフィードバック
   - hover 演出は `@media (hover:hover) and (pointer:fine)` でマウス端末限定にし、スマホで hover が残留する問題を回避

4. **文字・余白の流体スケール**: `clamp()` で見出しや余白が画面幅に応じて滑らかに伸縮。

5. **アクセシビリティの下地**: `prefers-reduced-motion` 対応、`:focus-visible`、`aria-pressed` / `aria-live` を実装済み。

> 既存コードにマージする場合は、(1) viewport メタタグ、(2) `.grid` の auto-fill 指定、(3) `.manaita-bar` のスマホ固定バー＋PC sticky 切替、の3点を移植すれば主要な効果が得られます。

## ② APIキーのセキュリティ構成

### 結論（推奨構成）

**GitHub Pages はそのまま＋Cloudflare Workers を1枚はさむ**のが、変更最小・無料・十分安全です。

```
ブラウザ(GitHub Pages) --食材名だけ--> Cloudflare Worker --APIキー付き--> LLM API
                                         ↑キーはWorkerのシークレットにのみ存在
```

- フロントのソース・リポジトリ・通信のどこにもAPIキーが現れない
- 無料枠: 1日10万リクエストまで（個人アプリには十分）
- Next.js 等への移行は不要。`fetch` 先のURLを Worker に向けるだけ

なぜ必要か: GitHub Pages は静的ホスティングなので、JSに書いたキーは**誰でも開発者ツールで読めます**。「難読化」「別ファイル化」「.envをビルドで埋め込む」はいずれも閲覧可能なため対策になりません。秘密を持てるのはサーバー側だけです。

### 実装手順（約15分）

1. Cloudflare アカウント作成（無料）
2. ターミナルで:
   ```bash
   npm create cloudflare@latest nokori-mono-proxy
   # → "Hello World" Worker を選択
   cd nokori-mono-proxy
   ```
3. 生成された `src/index.js` を同梱の `worker.js` の内容に置き換える
4. APIキーを**シークレットとして登録**（コードにもGitにも書かない）:
   ```bash
   npx wrangler secret put ANTHROPIC_API_KEY
   # プロンプトにキーを貼り付け
   ```
5. デプロイ:
   ```bash
   npx wrangler deploy
   # → https://nokori-mono-proxy.xxxx.workers.dev が発行される
   ```
6. `index.html` 内の `PROXY_URL` に発行されたURL（+ パス）を設定

### worker.js に入れてある防御策

- **CORS許可リスト**: `https://kenken6291.github.io` 以外のサイトからのブラウザ呼び出しを拒否
- **入力検証**: 食材は最大3件・各20文字までに制限（プロンプトインジェクション・浪費対策）
- **出力整形**: モデル応答をパースして name/desc のみ返却。フロント側でも `escapeHtml` でXSS対策済み

### さらに堅くしたい場合（任意）

- **レート制限**: Cloudflare ダッシュボードの Rate Limiting ルール（無料枠あり）で「同一IPから 1分10回まで」等を設定。Origin ヘッダーはブラウザ外からは偽装可能なので、濫用対策はレート制限が本命です。
- **Cloudflare Turnstile**（無料のCAPTCHA代替）をフロントに置き、Worker でトークン検証すると bot からの直接叩きをほぼ遮断できます。
- **コスト上限**: LLM側のダッシュボードで月額上限（ハードリミット）を設定しておくと、万一濫用されても被害額が固定されます。

### 代替案の比較

| 構成 | 変更量 | 費用 | 備考 |
|---|---|---|---|
| **Cloudflare Workers（推奨）** | 最小（fetch先変更のみ） | 無料枠10万req/日 | GitHub Pages継続可 |
| Vercel Functions | 小 | 無料枠あり | `api/suggest.js` を置くだけ。ただしホスティングごとVercelに寄せる方が自然 |
| Next.js 移行（Vercel） | 大 | 無料枠あり | 将来ユーザー認証・履歴保存等まで見据えるなら有力 |
| フロントにキー直書き | — | — | **不可**。難読化しても必ず抽出されます |

## ③ その他のUI/UX・機能改善の提案

**すぐ効くもの**

1. **選択上限時のフィードバック**: 3つ選択済みのとき、残りの食材ボタンを薄く非活性化（実装済み）。加えて「まな板がいっぱいです。どれかを外してください」というトーストを出すと迷いが減ります。
2. **検索ボタンの動的ラベル**: 「この3つで検索」を選択数に応じて「この2つで検索」に変える（実装済み）。0個時は disabled に。
3. **結果画面への自動スクロール**: スマホでは検索後に結果先頭へ `scrollIntoView` すると「押したのに何も起きない」と感じさせません。

**次の一手**

4. **選択の記憶**: `localStorage` に前回の選択を保存し、再訪時に「前回のまな板を再現しますか？」と提案。常備食材（卵・ご飯など）は再選択の手間が大きいので効果的です。
5. **レシピ辞書の拡充とスコア表示**: 「3つ全部使える」「2つ使える」をバッジで区別すると納得感が出ます。
6. **PWA化**: `manifest.json` と簡単な Service Worker を足すだけでホーム画面に追加でき、台所でアプリ的に使えます。ローカル辞書検索はオフラインでも動作します。
7. **買い足し1品の提案**: 「あと玉ねぎがあれば肉じゃがが作れます」のような“惜しい”レシピの提示は、このアプリのコンセプト（食材を最後まで使い切る）と相性が良いはずです。
8. **アレルギー・調理時間フィルタ**: 食材選択の下に「15分以内」「火を使わない」などのトグルを1行追加。

**品質面**

9. **OGP設定**: `og:title` / `og:image` を設定するとSNS共有時の見栄えが大きく改善します。
10. **計測**: 無料の Cloudflare Web Analytics（cookie不要）を入れると、どの食材がよく選ばれるかが分かりレシピ辞書の優先順位付けに使えます。
